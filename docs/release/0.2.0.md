# Orpheum 0.2.0 Release Notes

## Summary

Orpheum 0.2.0 is a minor release focused on onboarding reliability, clearer project state, stronger decision capture, and higher-confidence semantic review in discovery and planning workflows.

## Highlights

- `orpheum init` now persists repo-local catalog configuration in `.codex/orpheum/config.json`, writes a repo-root `ORPHEUM.md`, and reports richer setup metadata in JSON output.
- `orpheum doctor --json` now gives recovery-oriented setup information even before a project has an active `.orpheum` session.
- Artifact status now uses a stable persisted enum: `pending`, `present`, `verified`, and `failed`.
- Checks that are not executable in the current v1 model now surface as `not_evaluable_in_v1`.
- Discovery and planning scenarios now require a semantic artifact review phase and explicitly instruct the agent to use Planning Mode or the host environment's nearest equivalent while that review is in progress.
- Planning and handoff artifacts now have built-in sections for locked decisions, semantic review findings, and cross-artifact reconciliation so important architecture corrections do not stay trapped in chat context.

## Upgrade Notes

- Existing projects can rerun `orpheum init` to install the refreshed local skill, persist repo-local catalog configuration, and generate `ORPHEUM.md`.
- Consumers that parse `status --json` or check output should update for the new artifact-status values and the `not_evaluable_in_v1` check status.
- Host integrations that derive scenario-complete guidance should now expect semantic artifact review to be part of discovery/planning closeout for:
  - `project-discovery`
  - `project-planning`
  - `delivery-slice-planning`

## Validation

- `cargo fmt`
- `cargo test -p orpheum -- --nocapture`
- `cargo install --path crates/orpheum-cli --force`
